# Style Guide

## Audience

- 面向未来的自己或会直接复用项目的人，默认是半新手，不是完全零基础。
- 只解释会影响复现、判断或排错的概念。

## Writing Goals

- 写成可复现的操作教程，不写成概念综述。
- 优先保留真实步骤、真实命令、真实坑和真实决策。
- 长度尽量克制，只保留完成任务所需的信息。

## Structure Rules

- 文章开头必须有目标项目兼容的 frontmatter。
- 标题清晰，直接说明主题和结果。
- 开头 1 段，2-4 句，说明问题场景、目标和结果。
- `核心概念` 保留 2-4 个必要术语。
- `你需要准备的内容` 列前提与时间。
- 主体步骤统一使用 `## Step N: 动作 + 结果`。
- `下一步` 给 2-3 个延伸动作。
- `故障排除` 只保留最常见、最值钱的问题。

## Formatting Rules

- 步骤标题用编号；步骤内子项用无序列表，不用编号列表。
- 命令和路径用反引号；多行命令用代码块。
- UI 文案用粗体。
- 不单独列 `Sources`；必要链接放首次提及时内联。

## Tone Rules

- 保持直接、紧凑、可信。
- 不用“你完全不需要懂任何东西”这类过度安抚语气。
- 允许默认读者知道基本终端、文件、仓库概念。
- 遇到关键分支，写出为什么选当前方案。

## Fidelity Rules

- 不把没有发生过的步骤写成既定事实。
- 如果某一步来自推断或补充研究，明确点出。
- 研究只用于刷新时效性信息，不用于覆盖真实经历。
